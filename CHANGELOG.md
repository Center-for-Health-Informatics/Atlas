- **[atlas] Bumped `jquery` 1.12→4.0, phase 3 step (d) of the dependency upgrade roadmap.** A research pass first (grep for every removed/deprecated jQuery 3.0/4.0 API, jQuery UI widget usage, DataTables' declared jQuery peer range) came back reassuringly clean: none of jQuery 3.0/4.0's actual API removals (`$.isArray`, `$.parseJSON`, `.size()`, `.pipe()`, `$.fn.andSelf()`, etc.) are used anywhere in app code, no IE-specific code paths exist to be affected by jQuery 4 dropping IE support, and DataTables/jQuery UI both declare (and, mostly, actually have) jQuery-4-compatible ranges — so this was expected to be, and mostly was, a low-risk bump relative to the Bootstrap step.
  - **Two real `.then()`-on-raw-`$.ajax()` risk spots**, flagged by the research pass and fixed proactively rather than waiting to hit them: jQuery 3+'s Promises/A+-compliant Deferred means an unguarded `.then(onSuccess)` (no rejection handler, no `.catch()`) now silently swallows ajax failures *and* exceptions thrown inside the success callback, instead of surfacing them. `services/Vocabulary.js`'s `getDomains()` wrapped such a call in a `new Promise((resolve, reject) => ...)` but never wired `reject` to anything — under jQuery 4 this could leave the returned promise hung forever on any ajax failure. Fixed by passing `reject` as `.then()`'s second argument. `assets/ohdsi.util.js`'s `cachedAjax()` had the same shape but for a fire-and-forget cache-write side effect (lower severity — nothing awaits it) — added a no-op rejection handler to avoid an unhandled-rejection console warning.
  - **Modernized the 3 real `.bind()`/`.unbind()` call sites** (`Application.js`'s `beforeunload` handler, `assets/jqueryui/autoGrowInput.js`, `assets/jqueryui/jquery.ddslick.js`) to `.on()`/`.off()` — not required (jQuery 4 still supports both), but low-cost and removes latent risk if a future jQuery release drops them for real.
  - **Test suite broke immediately on the bump** — not from anything in the survey, but because jQuery 4 hard-throws `"jQuery requires a window with a document"` at *import time* in a windowless environment, where 1.12 loaded permissively. The Node test runner has no DOM at all, so any test importing a module that imports jQuery (`CommonUtils.test.js` → `utils/CommonUtils.js` → `jquery`) failed outright. Fixed the same way the existing test harness already handles Vite-only bare specifiers (`appConfig`, `atlas-state`, etc., per `tests/hooks.js`): added a `jquery` stub (`tests/stubs/jquery.js`, a `Proxy`-based infinitely-chainable no-op) to the module-resolution hook's alias map — safe because the actual `$(...)` DOM-manipulation call sites reached by `CommonUtils.js`'s tested functions are never exercised in these particular unit tests.
  - **The one genuine functional regression, caught only by live `run-atlas` interaction, not by grep or the build**: jQuery UI's `sortable` widget (used for drag-reordering criteria in the cohort builder's Inclusion Rule/Criteria Group editors) threw `Syntax error, unrecognized expression: unsupported pseudo: data` the moment a nested-sortable lookup ran. Traced to `sortable.js`'s own internal `:data(...)` pseudo-selector lookup (`this.currentItem.find(":data(" + this.widgetName + "-item)")`) — initially suspected to be jQuery 4's well-known Sizzle-selector-engine removal, but turned out to be simpler and unrelated to that: jQuery UI ships this pseudo-selector as its own separate, real module (`jquery-ui/ui/data.js`, ~15 lines, registers `$.expr.pseudos.data` via jQuery's still-fully-supported custom-pseudo extension point), which `sortable.js` declares as an AMD dependency (`define(["jquery", "./mouse", "../data", ...])`) — but since it's a classic UMD script rather than a real ES module, Vite never resolves that declared dependency automatically, so it was on `js/ko-sortable-setup.js`'s explicit side-effect-import list to include it, and it simply never had been. Fixed with one added import line (`import 'jquery-ui/ui/data'`) — no monkey-patch, no library replacement, just wiring in the official module jQuery UI itself ships for exactly this purpose. Verified fixed live: the cohort builder's "Add attribute" → numeric-operand editor flow, which exercises this exact nested-sortable code path, now completes without error.
  - Verified with `npm run lint` (clean), `npm run build:dev` (clean, no new warnings), the full test suite (34/34, after the jQuery test-stub fix), and `run-atlas` smoke testing: cohort builder attribute-editor dropdowns and the sortable-dependent "Add Age at Occurrence" flow work correctly post-fix; the Profiles page (jQuery UI `draggable`-modal territory) loads cleanly. `npm audit` dropped to 8 vulnerabilities (from 9). `npm outdated`'s major-version list is now down to just `datatables.net`(+`buttons`+`select`) — step (e), the last item in the Phase 3 bucket — plus excluded `eslint`.
