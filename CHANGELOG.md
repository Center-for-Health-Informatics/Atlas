# Changelog

Notable fixes made while shaking out the Atlas + webapi rewrite. Not exhaustive version history — just a running log of bugs found and fixed during testing, newest first. Each entry notes which repo (`atlas` or `webapi`) it touched.

## 2026-07-08

- **[atlas] Cohort Definition builder/viewer completely inert for ~53 components — biggest bug found this session.** Discovered while smoke-testing the "New Cohort" flow: it crashed with `Cannot read properties of undefined (reading 'prototype')`. Root cause: every criteria component under `js/components/cohortbuilder/components/` and `js/components/cohortdefinitionviewer/components/` (Condition/Drug/Procedure/Visit/Measurement/Death/etc. — every clinical event type, plus the group/inclusion/end-strategy editors) defined a real Knockout view model class or function, but the file's export only included the template: `export default { template }`, missing `viewModel: TheClassName`. Registered via `ko.components.register(name, importedModule)`, this means Knockout renders the template with **no view model at all** — every `self.foo` property, computed, and method the class sets up silently doesn't exist. It only threw where something explicitly dereferenced the (undefined) view model class, like `commonUtils.build()`'s `viewModelClass.prototype`; everywhere else it would have just rendered inert templates with `data-bind` expressions throwing `ReferenceError` the moment a user interacted with them (e.g. `helpCohortEventsOpened is not defined`, hit next while testing). Wrote a small script to find every file matching this shape (`grep -rl "^export default { template }$"`), auto-detect the correct view-model class/function name (last top-level `function`/`class` declaration in the file — verified this heuristic against every match first), and fix the export to `{ viewModel: Name, template }`. Verified project-wide afterward that no instances of the pattern remain. This was the actual blocker for ever exercising Cohort Definitions, Estimation, Prediction, or Characterizations end-to-end, since all four build on cohort-definition criteria editors.
- **[atlas] jQuery UI `autocomplete` and `datepicker` bindings crashed with `$(...).appendTo(...).menu is not a function`** — found immediately after the fix above, still inside the cohort-definition builder (the "before/after event index date" day-count autocomplete). `jquery-ui/ui/widgets/autocomplete.js` is a UMD module whose non-AMD "browser globals" fallback just calls `factory(jQuery)` directly, assuming its sibling widgets (`menu`, `keycode`, `position`, `unique-id`) were already loaded as separate `<script>` tags patching the shared `$.ui`/`$.fn` namespace — true under the old RequireJS shim config, not automatically true under Vite/ESM where only the files someone explicitly `import`s get evaluated. Fixed by adding the missing side-effect imports (`jquery-ui/ui/keycode`, `jquery-ui/ui/position`, `jquery-ui/ui/unique-id`, `jquery-ui/ui/widgets/menu`) to `autocompleteBinding.js`, and `jquery-ui/ui/keycode` to `datepickerBinding.js` (same dependency, not yet hit live but same latent bug on inspection).
- **[atlas] Confirmed working end-to-end after the fixes above**: created and saved a real Cohort Definition (name + Condition Occurrence entry criteria), and opened the "New" editor for Estimation, Prediction, and Characterizations — all render fully and reuse saved cohort definitions via picker modals (Prediction's "Add Target Cohort" and Characterizations' "Import" cohort-definition modal both correctly listed the cohort just created). Workflows documented in `USAGE.md`.
- **[atlas] Minor, not fixed (out of scope for this repo)**: the "Cohort Exit" panel shows a raw, untranslated i18n key (`components.endStrategyEditor.eventWillPersistUntil2`) next to the event-persistence dropdown, and the "created by X on Y" authorship label has a `cratedDate` typo in its i18n template — both are server-side translation strings served by WebAPI's `/i18n` endpoint, not part of this repo.

- **[atlas] Swept the whole codebase for the same class of bug as the `d3-labeler` fix below and fixed ~60 files.** `npm run lint` already had the evidence: 248 `no-undef` errors, all from the same two AMD→ESM conversion gaps — (1) code that used globals RequireJS used to shim in (`$`/jQuery in ~37 files, `_`/lodash in ~10, plus `saveAs`, `Prism`, `moment`, `qs`) without a real `import`, and (2) bare `identifier = value` assignments that created implicit globals under non-strict AMD scripts but throw `ReferenceError` under ES modules' mandatory strict mode. Added the missing imports throughout; for `Prism`, added `js/utils/prism.js` to explicitly set `window.Prism` since `prismjs`'s bundled language plugins (e.g. `prism-sql.js`) expect it as a real global and ESM module evaluation order means the assignment must live in its own dependency module, not inline before the plugin import. Also deleted the fully-unreferenced vendored `js/require.js` (RequireJS itself, dead since the ESM migration).
  Beyond the mechanical import fixes, this surfaced several real correctness bugs that had been silently broken since the migration (all masked the same way — valid syntax, only throws when the code path actually runs):
  - **`AuthAPI.js`'s `isPermitted()`** — the core permission-check function used app-wide — assigned to an undeclared `firstPerm` instead of `const firstPerm`. Every single permission check would have thrown.
  - **`pages/Route.js`** threw `new Exception(...)` (not a real global) instead of `new Error(...)` in the base `handler()` override guard.
  - **`services/file.js`**'s `_makeRequest()` referenced a `reject` that only existed in the *callers'* Promise executors; on any XHR error the zip-download/export promises would hang instead of rejecting. Threaded an `onError` callback through properly.
  - **`pages/cohort-definitions/routes.js`** — the conceptsets route referenced a bare `ConceptSetStore` that was never bound (only dynamically imported and discarded in a `Promise.all` array); fixed by destructuring `{ default: ConceptSetStore }` from the resolved import. A separate route handler also did bare `qs = router.qs()` instead of `const qs = ...`.
  - **`components/conceptset/utils.js`** called `loadSourcecodes` (lowercase c) when the function is defined/exported as `loadSourceCodes` — simple casing typo. Also deleted `removeConceptsFromConceptSet`, a fully unreferenced dead function whose body called a nonexistent `resolveConceptSetExpression()` (the real one is a method on `ConceptSetStore` instances, not a bare function).
  - **`pages/estimation/cca-manager.js`** — two `addConceptSetToEstimation()` calls for `positiveControlSynthesisArgs` covariate settings referenced a loop `index` from a `forEach` that had already ended (dead scope), and both passed `.targetName.includedCovariateConcepts` instead of `.propertyName.{included,excluded}CovariateConcepts` (copy-paste from the array-based branches above it).
  - Smaller ones: `circe/ConceptSetBrowser.js` had a `config`/`appConfig` typo; `FeasibilityReportViewer.js` called bare `viewerWidget(self)` instead of `params.viewerWidget(self)`; three `cohortdefinitionviewer` components (`DrugEra`, `DrugExposure`, `VisitOccurrence`) were missing the `components/cohortbuilder/utils` import their sibling components all have; `ConceptPicker.js`'s `search()` had the same bare-assignment bug as `routes.js`; `atlas.cohort-editor.js`'s `onGenerate`/`getExpressionJSON` methods were dead code (never bound in the template, never called) referencing nonexistent `pollForInfo`/`pruneJSON` — deleted rather than reconstructed, since there was no way to know the intended behavior.
  Verified: `no-undef` count is 0 (was 248), `npm run build` and `npm test` (18/18) still pass, dev server cold-starts cleanly.
- **[atlas] Dev server crashed on cold dependency scan: `Cannot assign to import "labeler"`** — `js/extensions/d3-labeler.js` was a mechanical AMD→ESM conversion of a vendor UMD plugin that originally patched the global `d3` object (`d3.labeler = function () {...}`); the converted version did `import * as d3 from 'd3'; d3.labeler = ...`, which esbuild rejects because namespace imports are immutable. This had gone unnoticed because Vite's cached dep-optimization skipped re-scanning it, until a lockfile change (`npm install`) forced a fresh scan. Also found: the function body relied on several undeclared-global assignments (`rej`, `w_orient`, `energy`, `mcmove`, `mcrotate`, `intersect`, `cooling_schedule`, `theta`) that only worked in the old non-strict AMD/global-script context — ES modules are always strict mode, so these would have thrown `ReferenceError` the first time the profile chart's label-overlap simulated annealing actually ran. Fixed by exporting `labeler` as a named function instead of mutating `d3`, declaring all the previously-implicit variables, and updating the sole caller (`js/extensions/bindings/profileChart.js`, used by the patient profile chart) to `import { labeler } from 'extensions/d3-labeler'` and call `labeler()` directly.
- **[atlas] Production build was silently broken; now fixed** — `build`/`build:dev`/`build:docker` shelled out to `build/optimize.js` (RequireJS optimizer `r.js` + Babel + Terser), which exited 0 but produced a ~9K-line bundle containing only 3 `define()`-wrapped modules — it couldn't parse the ESM app source and failed silently. A Docker image built from `main` today would have shipped a broken frontend with no build-time error. Repointed `build:dev` at `vite build --config vite.config.mjs` (already proven to work via the dev server config) and deleted `build/optimize.js` and `build/polyfill.js` (superseded by `@vitejs/plugin-legacy`).
- **[atlas] `genversion` was emitting an AMD-wrapped `js/version.js`** — found while verifying the build fix above: the old `genversion -s js/version/version.js && r_js -convert js/version js/` pipeline produced `define(function (require, exports, module) { module.exports = '...' })`, which the ESM app can't `import { version } from 'version'` — this had been masked by `build/optimize.js`'s silent failure. Replaced with `genversion -e -s js/version.js`, which writes plain ESM directly.
- **[atlas] Dropped now-unused build devDependencies** — `requirejs`, `@babel/core`, `@babel/preset-env`, `@babel/plugin-transform-object-rest-spread`, `@babel/polyfill`, `terser`, and `html-document` were only used by `build/optimize.js` and the old `genversion`/`r_js -convert` step, both gone now.

## 2026-07-07

- **[atlas] Dropped Jest for Node's built-in test runner** — `npm test` now runs `node --test` instead of Jest. The old setup depended on a RequireJS shim (`jest.config.js`'s `requireAmd`/`requirejsInstance`) to load AMD-style modules, which no longer made sense once the app source moved to ESM (it was already failing 18/18 tests — see `MIGRATION_STATUS.md`). Ported all 5 existing test files (`AutoBind`, `BemHelper`, `CommonUtils`, `DataTypeConverterUtils`, `ExceptionUtils`) to plain ESM imports + `node:assert/strict`; added `tests/hooks.mjs` (a small module-resolution hook, registered via `tests/register-hooks.mjs`) to stand in for the handful of Vite-only bare-specifier aliases (`appConfig`, `atlas-state`, etc.) that `CommonUtils.js` imports. Also updated the one stale assertion in `CommonUtils.test.js` that had encoded the `isSelected`-stripping bug fixed above — it now asserts the corrected, intended behavior instead. `jest` and `jest.config.js` are gone; `package.json` gained `"type": "module"`.

- **[atlas] Concept set save silently reported success on name conflict** — `saveConceptSet()` in `js/pages/concept-sets/conceptset-manager.js` showed an "already exists" alert but didn't actually communicate failure to its caller. `save()` then unconditionally reset the dirty flag afterward regardless of outcome, so the cart nav icon turned green and the Save button disabled itself as if the concept set had been persisted — even though nothing was saved and the set was still unnamed/id 0. Fixed by having `saveConceptSet()` return `true`/`false` and only resetting the dirty flag on an actual successful save.

- **[atlas] Duplicate concepts silently added to concept sets** — `getSelectedConcepts()` in `js/utils/CommonUtils.js` stripped the `isSelected` observable off returned concepts before handing them back. That broke `clearConceptsSelectionState()`'s attempt to uncheck a concept's checkbox after "Add To Concept Set" (it operated on the stripped copies, not the live observables), so the checkbox silently stayed checked. Any subsequent Preview or Add re-added the same concept on top of what was already in the set, since Atlas has no dedup check. Fixed by no longer stripping `isSelected`.

- **[atlas] Unhandled 422 fetching GitHub release-notes on the Home page** — `home.js` unconditionally fetched closed issues from `api.github.com` for release notes, defaulting a missing `milestoneId` to `'*'` ("any milestone"). Since webapi's `/info` stub always returns `milestoneId: null`, this turned a small scoped query into an unbounded fetch of every closed issue in OHDSI/Atlas history, which GitHub's paginated Issues API now hard-rejects (422) past ~1000 results via `page`. Fixed by skipping the GitHub fetch entirely when no real milestone is available, and wrapping the call in try/catch so a GitHub failure can no longer break the Home page.

- **[webapi] Cohort Pathways browser crashed on load** — `GET /pathway-analysis` returned a bare array, but Atlas's `pathways` browser (and `cohort-pathway-browser.js`) always expects a paginated `{ content: [...] }` shape (matching how `reusable.js` already worked). Mismatch caused `coalesceField`'s `list.forEach` to throw on `undefined`. Fixed by passing `{ paginated: true }` to `makeAnalysisRouter('pathway_analysis', ...)` in `src/routes/pathway.js`.

## Notes

- Atlas and webapi are separate repos/containers. A fix under `[webapi]` requires a container rebuild + recreate to take effect (`docker compose build` + recreate) — restarting Atlas alone, or just refreshing the browser, does nothing for webapi-side changes.
- A fix under `[atlas]` just needs a browser refresh (dev server) or a rebuild of the served bundle, depending on how Atlas is being served.
