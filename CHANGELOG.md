# Changelog

Notable fixes made while shaking out the Atlas + webapi rewrite. Not exhaustive version history — just a running log of bugs found and fixed during testing, newest first. Each entry notes which repo (`atlas` or `webapi`) it touched.

## 2026-07-10

- **[atlas] The Docker image had never actually been shipping the real production bundle — likely since the AMD→ESM migration.** Found while modernizing `Dockerfile`'s base images. `Dockerfile`'s final nginx stage copied `/code/index.html` (the raw *source* file, used only as Vite's build entry) and the raw `/code/js` source tree straight into the served image, instead of Vite's actual build output (`js/assets/bundle/`, per `outDir` in `vite.config.js`). A real browser can't execute that raw source — it uses bare ESM specifiers (`import sharedState from 'atlas-state'`) that only resolve via Vite's build-time alias config. Two more bugs had to be fixed just to get a real `docker build` to succeed at all: `vite.config.js` was never `COPY`'d into the builder stage (`Could not resolve "/code/vite.config.js"`), and `index.html` was copied *after* `RUN npm run build:docker` instead of before, so Vite had no entry module to build (`Could not resolve entry module "index.html"`) — both suggest this Dockerfile hadn't been run end-to-end since before the migration. Fixed the copy strategy: the final stage now copies the *contents* of `js/assets/bundle/` (Vite's real `outDir`) directly into the `atlas/` docroot, matching `base: '/atlas/'` in `vite.config.js`, which expects asset URLs like `/atlas/assets/index-XXXX.js`. Also dropped the `node_modules` copy into the final image — confirmed (via grep of both source and the real built bundle) that nothing references it at runtime anymore; it was serving pre-bundling-era UMD vendor files directly, which Vite now bundles instead.
- **[atlas] A second latent bug the above fix exposed: `images/` was never copied into the Docker *builder* stage, only the final stage.** `index.html` references `favicon.ico`/`atlas_loading.svg`/`ohdsi_color.png` by relative path; without the source files present in the builder, Vite silently left those references untouched instead of hashing/bundling them (no build error) — worked in every local test because `vite build` was run from the real project root where `images/` exists alongside `index.html`, masking the bug until a byte-for-byte container rebuild. Fixed by adding `COPY ./images /code/images` to the builder stage. Separately confirmed (and preserved) that the *final*-stage `images/` copy is still independently required for a different reason: `js/components/circe/components/GenerateComponentTemplate.html` references `images/running.png` via a runtime-loaded raw HTML string that Vite's asset pipeline never sees.
- **[atlas] Modernized `Dockerfile`/`docker/` for the first time since before the migration.** Builder base image `node:18.14.1-alpine` (EOL) → `node:lts-slim` (Debian, floating tag per team preference). Production base `nginxinc/nginx-unprivileged:1.27.2-alpine` → `1.28.0-bookworm` (Debian, digest-pinned). Added explicit `ENV NODE_ENV=production` in the builder, placed *after* `npm install` (vite is a devDependency; setting it before would make npm skip installing it) and before the build step, rather than relying on `vite build`'s implicit default. Added `.dockerignore` entries for `js/assets/bundle`/`fonts`/`images`/`version.js` — a stale local build artifact leaking into the build context (`.dockerignore` didn't exclude it) is exactly how the `images/`-in-builder bug above went undetected in earlier local testing.
- **[atlas] Container runtime configuration (`WEBAPI_URL`, `ATLAS_INSTANCE_NAME`, all `ATLAS_SECURITY_*` flags, etc.) had been completely non-functional since the ESM migration.** `docker/config-local.js` (still AMD `define([], ...)`-wrapped) was `envsubst`'d and copied into the image at container start, matching the pre-migration RequireJS-era mechanism where it was fetched at runtime — but the ESM app only ever consumed local config via a *build-time* static `import localConfig from 'config-local'`, which Vite inlines during the builder stage. Nothing in the shipped app ever fetched the runtime-substituted file, so `docker run -e WEBAPI_URL=...` silently did nothing, for an unknown period. Fixed by adding a real runtime-config path: `js/runtime-config.js` (new, tracked, default `window.ATLAS_RUNTIME_CONFIG = {}`) loaded via a plain `<script>` tag in `index.html` *before* the `type="module"` entry script (classic scripts run immediately in document order; module scripts are deferred, so this ordering is guaranteed); `js/config.js` merges `window.ATLAS_RUNTIME_CONFIG` in last, giving it final precedence over build-time config. `docker/config-local.js` → renamed `docker/runtime-config.template.js`, wrapper changed from AMD `define()` to a `window.ATLAS_RUNTIME_CONFIG = (function(){...})()` global assignment. Verified end-to-end with a real `docker build` + `docker run -e WEBAPI_URL=... -e ATLAS_INSTANCE_NAME=...`: the served `runtime-config.js` correctly reflects the env vars, with no rebuild between different `-e` values.
- **[atlas] `docker/30-atlas-env-subst.sh` switched from `envsubst` to a `sed`-based substitution loop.** Needed because the runtime-config template had to become genuinely valid, lint-clean JS (`neostandard`), and `envsubst`'s `${VAR}` placeholder syntax is indistinguishable from a real JS template-literal interpolation — forcing an ugly choice between quoted-string placeholders (trips `no-template-curly-in-string`/`no-constant-condition`) or real backticks (then every placeholder is a genuinely undefined JS identifier reference, i.e. the file would `ReferenceError` if executed before substitution — `no-undef` correctly catches this). Switched the placeholder syntax to `@@NAME@@` markers (no `$`, zero collision with JS) and wrapped every flag comparison in an `isTrue(value)` helper so comparisons are call expressions rather than literal-vs-literal comparisons, which `no-constant-condition` doesn't flag. The entrypoint script now extracts marker names actually present in the file (`grep -o`) and substitutes each via `printenv` (an earlier `eval`-based indirection attempt hit a "Bad substitution" error under `dash`, the `/bin/sh` on Debian bookworm). Result: `docker/runtime-config.template.js` passes `eslint` with zero errors and zero suppressions.
- **[atlas] Collapsed three build-time config files into one.** Per team preference for single env-var-driven config (no separate override files to create/merge): deleted `js/config-local.js` and `js/config-gis.js` (the latter was already empty/unused) and their `vite.config.js` aliases. `js/config.js` now reads `import.meta.env.VITE_WEBAPI_URL`/`VITE_WEBAPI_NAME` directly (Vite's native env-var passthrough — picks up real shell/`direnv` env vars, no `.env` file required, though a gitignored `.env.local` also works). Updated `CLAUDE.md`, the `run-atlas` skill's setup instructions, and `js/config/app.js`'s comments accordingly.
- **[atlas] `<script src="/js/runtime-config.js">` used an absolute domain-root path, breaking under any reverse-proxy subpath deployment** (e.g. `chi-tools.uc.edu/atlas/` — the browser requested `chi-tools.uc.edu/js/runtime-config.js` instead, a 404). Found via a real staging deployment behind an nginx reverse proxy — every earlier local/container test happened to request paths directly under `/atlas/`, masking the bug. Fixed using Vite's `%BASE_URL%` HTML env-replacement (`<script src="%BASE_URL%js/runtime-config.js">`), which bakes in the correct `/atlas/js/runtime-config.js` at build time — confirmed Vite leaves this plain (non-module) script tag's path otherwise untouched (no fingerprinting), so the filename stays stable for the entrypoint script to rewrite.
- **[atlas] Production builds silently swallowed *all* fatal boot errors, making the above (and any future) production-only failure completely invisible.** `js/main.js`'s top-level `run().catch(err => { if (debug) { console.error(...) } })` only logged when `NODE_ENV !== 'production'` — same gate as the routine step-by-step load tracing, but incorrectly also covering the *fatal* top-level catch. In production, any of `run()`'s ~30 dynamic imports (or `Application.bootstrap()`) throwing meant total silence: no console output, no title change, just an indefinitely stuck splash screen. Fixed by always logging the top-level `console.error`, while leaving the step-tracing itself dev-only as before (that gating was intentional, not the bug).
- **[atlas] `TypeError: t.i18n is not a function`, hanging the app on the splash screen in production only (dev server fine) — a `ko.i18n` registration-order bug only exposed by Rollup's production chunking.** `js/const.js` calls `ko.i18n(key, defaultValue)` at module top level (eagerly); `ko.i18n` is only defined as a side effect of importing `js/extensions/bindings/i18nBinding.js` elsewhere (`extensions/bindings/main.js` even has a comment: "i18nBinding must be first — it defines ko.i18n used by const.js and others"). That ordering guarantee held only because `main.js`'s `run()` happens to `await import('databindings')` before `await import('const')` — an *implicit* dependency on an unrelated file's import order, not a direct one. Confirmed via `npm run dev` (native unbundled ESM — works perfectly, no error) vs. the production build (crashes) that this is purely a Rollup chunking artifact: `const.js` gets pulled into a shared chunk (dozens of page modules depend on it), and that chunk's evaluation order doesn't respect the sequential-await ordering `main.js` relies on. Reproduced identically in both Chromium and Safari, so not a browser-engine quirk. Fixed by making the dependency explicit where it's actually needed: added `import './extensions/bindings/i18nBinding'` directly to the top of `js/const.js`. Verified in a real production Docker build with a real WebAPI: full app shell renders (nav sidebar, all menu items), past the splash screen.
- **[atlas] `README.md` rewritten with Development/Configuration/Docker sections** reflecting the current npm scripts and the two env-var-driven config layers (build-time `VITE_*`, container-runtime `WEBAPI_URL`/`ATLAS_*`) — previously just the generic upstream OHDSI project blurb with no build/run/config instructions specific to this fork.

- **[webapi] `GET /tag/` and `POST /tag/` never returned `createdDate`/`createdBy`, even though the `tag` table has always stored both.** Found driving Tag Management: the "Created" column literally rendered the text `false` for every row. Root cause: `atlas/js/utils/DatatableUtils.js`'s `getDateFieldFormatter('createdDate')` returns its `defaultValue` argument (which defaults to the literal boolean `false`, not an empty string) whenever the field is absent from the row — and that return value is rendered directly as cell text, not treated as a boolean flag. `webapi/src/routes/tags.js`'s `rowToDto()` simply never surfaced `created_by`/`created_date` from the row. - **[webapi] Tag Management's group→tag hierarchy didn't exist server-side at all — creating a tag "in" a group silently created an unrelated top-level group instead.** The `tag` table had no parent/child relationship, and `rowToDto()` never returned a `groups` field. Atlas's `tag-management.js` treats any tag with no `groups` as a group itself (`!t.groups || t.groups.length === 0`) and filters `allTags()` for `t.groups[0].id === group.id` to build the "Tags in Group" list — so a tag created without a `groups` array both vanished from its intended group's tag list *and* showed up as a second bogus top-level group. Also found in the same investigation: color/icon/description and the four group-editor checkboxes (mandatory/showGroup/multiSelection/allowCustom) were accepted by the create/edit forms but never persisted at all (no DB columns), which is why every group rendered with the default gray `#cecece` swatch regardless of what color was picked. Fixed with migration `009_tag_groups.sql` (adds `parent_id`, `description`, `color`, `icon`, `mandatory`, `show_group`, `multi_selection`, `allow_custom`, `protected` to `tag`) and rewrote `POST/PUT /tag/:id` to persist all of it and `rowToDto()` to return `groups: [<parent tag DTO>]` for child tags (needed both for group-membership filtering and because `tag-management.js` reads `d.groups[0].color`/`.icon` as a rendering fallback). Verified live: created a "Demographics" group with a custom blue color, created a "Female" tag inside it, and the tag correctly appears under "Tags in Group 'Demographics'" wearing the group's color — reloading the page preserves all of it.
- **[webapi] Cohort Pathways generation was a flat 501 stub — `POST /pathway-analysis/:id/generation/:sourceKey` always returned "Not Implemented", so the Executions tab's Generate button was a dead end.** Audited whether this needed CIRCE/ARACHNE like Estimation/Prediction and concluded no: target and event cohorts are references to *already-generated* cohorts (same reasoning that made IR tractable), so pathway-finding is just per-person event ordering against `${resultsSchema}.cohort`. Implemented real execution in new `webapi/src/pathway-generation.js`: for each target cohort, anchors each person at their earliest target-cohort entry, collects event-cohort occurrences on/after that date, combines same-window events into a single step via bitwise-OR (matching the `code = 2^i` per event-cohort-index encoding `pathway-results.js` expects for its sunburst chart), optionally collapses consecutive repeat steps (`allowRepeats`), truncates to `maxDepth`, and aggregates person counts per unique dash-joined path — with `minCellCount` suppression. Wired up as real (not stubbed) `POST/DELETE /:id/generation/:sourceKey`, `GET /:id/generation`, and `GET/DELETE /generation/:generationId(/result|/design)` routes in `webapi/src/routes/pathway.js`, backed by a new `pathway_generation` table (migration `010_pathway_generation.sql`) that caches each run's computed report as JSON since Atlas's `PathwayService` addresses a single execution by an opaque `generationId`, unlike IR's `(analysisId, sourceKey)` keying. Had to register these via `makeAnalysisRouter`'s `configure` hook rather than appending routes after the router is returned — the factory registers its own generic 501/404 generation stubs on the same router object, and Express dispatches to whichever matching route was registered first, so routes added afterward would have been silently shadowed. Documented simplifications (single earliest-entry anchor per person, small-cell paths dropped rather than bucketed) directly in the module, same spirit as `ir-generation.js`'s stratification TODO. Verified live end-to-end against the `testing` source: generated "Test Pathway QA" (2 target/event cohorts, already saved from a prior session), got a real completed execution in the Executions tab, and both the sunburst Visualization and Tabular report views rendered correct person counts (1,152 target cohort, 100% pathway coverage, e.g. 993 persons following the "testing · RRI → Test Cohort Documentation" step sequence) with no console errors.
