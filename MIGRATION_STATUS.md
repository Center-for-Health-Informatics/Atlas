# AMD → ESM Migration Status

There's no single doc that's tracked this as it happened — the real history lives across git log messages, code comments, and (until now) a Claude memory file nobody else could see. This is an attempt to consolidate it, last updated 2026-07-08. Verify against current code/git log before trusting specifics — this will drift.

## Where things stand

**Source conversion: essentially done.** Of ~785 JS files under `js/`, only 9 non-build-output files still use `define(...)` — and all 9 are third-party/vendor code, not app source:
- RequireJS plugin shims: `js/extensions/plugins/{text,css.min,less,optional}.js`
- Vendor UMD libs: `jquery.ddslick.js`, `d3.slider.js`, `d3-scale-chromatic.1.3.0.min.js`, `jnj.chart.js`
- `js/extensions/bindings/knockout.selectOnFocus.js`

Git history for this: `3ab53419` (phases 0 & 1), `a7cd1b94` (phase 2: batch-converted 620+ files), `5c76a6a9` ("mid AMD→ESM migration"), plus assorted individual bug fixes since.

**Dev server: works.** `npm run dev` runs Vite at whatever host/port is configured (see `vite.config.js`), proxying `/WebAPI` to the webapi container. App boots, renders nav, and — as of 2026-07-08 — Home, Vocabulary Search, Concept Set creation/editing, Cohort Definitions (create/add-criteria/save), and the "New" editors for Estimation, Prediction, and Characterizations have all been smoke-tested end-to-end with a real browser (Playwright) against the `https://dev.lastchance.pub/webapi/` test instance, with bugs found and fixed along the way (see `CHANGELOG.md` — includes a big one: ~53 cohort-builder/viewer components had no view model wired up at all). Workflows for all of these are now documented in `USAGE.md`. As of 2026-07-08, ACHILLES has been run against the Synthea CDM (`GET /cdmresults/testing/dashboard` returns real person counts/age distributions, not empty data), so Data Sources dashboards are populated. Cohort Definition Generation/Execution against the `testing` source now works too (was blocked until 2026-07-08 by an unrelated mssql connection-string-parsing bug in `cohortdefinition.js`, since fixed — see `webapi`'s own history), and real Incidence Rate analysis execution was implemented the same day.

Later on 2026-07-08, a second testing pass (using the new `run-atlas` skill — a headless-Chromium Playwright driver, see `atlas/.claude/skills/run-atlas/`) covered the remaining major workflows, specifically exercising the save-then-reload-by-fresh-navigation path (the two worst bugs of the day, both `/check` 404s and a broken IR reload, only showed up that way — a curl-only check of the same endpoints missed them). Verified end-to-end with no console errors: **Cohort Pathways** (create/save/reload by id — found and fixed a missing `POST /pathway-analysis/check` stub), **Reusables** (create/save/reload by id — clean, no bugs), **Characterizations** (save/reload by id — found and fixed a missing `POST /cohort-characterization/check` stub), **Prediction** (save/reload by id — found and fixed a missing `POST /prediction/check` stub), **Estimation** (save/reload by id — clean, confirms the earlier `estimation/check` fix holds), **Profiles** (found and fixed three stacked bugs: `GET /:sourceKey/person/:id` 500'd from reading the wrong `sources.js` accessor, then returned empty data from reading UPPER_CASE mssql column names that don't exist since mssql returns exact-case-as-written lowercase, then the chart rendered as garbled `NaN`-positioned SVG because `startDay`/`endDay` day-offset computation was long-dead commented-out code — see `CHANGELOG.md` for all three), **Jobs** (list renders correctly against real job history; "Job Name" showing "n/a" is Atlas's own intentional fallback for jobs whose stored params don't include a `jobName` key, not a bug), **Configuration** (source list, Tag Management, Run Diagnostics all clean), and **Tools** (found and fixed a raw-i18n-key page title, `common.tools` missing its default-text argument; Add Tool flow verified end-to-end). See `CHANGELOG.md` (2026-07-09) for full details and regression tests. Estimation/Prediction *execution* remains deliberately out of scope — would require the ARACHNE execution engine, judged not worth standing up for a single local instance.

**Production build: fixed as of 2026-07-08 — but that only covered `vite build` itself, not the Docker deployment path.** `build:dev` now runs `vite build --config vite.config.js` directly; `build` and `build:docker` compose on top of it unchanged. `build/optimize.js` (the old RequireJS optimizer + Babel + Terser path) and `build/polyfill.js` (only used by it, superseded by `@vitejs/plugin-legacy`) are deleted. The `compress` script (a separate `terser` pass) is gone too — `vite build` already minifies. Verified at the time: `npm run build`, `npm run build:docker`, and `npm test` all pass; the resulting `js/assets/bundle/` is a full chunked build (1264 modules transformed, `index.html` references real hashed asset chunks) and was smoke-checked via `vite preview`.
  - **Correction, 2026-07-10:** `vite build`/`vite preview` working was never the same claim as "the Docker image serves working production output" — and it didn't. `Dockerfile` had never actually been run end-to-end since before the migration (missing `COPY ./vite.config.js`, wrong `index.html` copy order — both fixed 2026-07-10) and, once it *did* build, its final stage was copying the raw source tree into the served image instead of Vite's real `js/assets/bundle/` output — meaning every Docker-deployed Atlas was shipping unexecutable raw ESM source, not the bundle `vite preview` had verified. Also found and fixed a genuine production-only app bug that `vite preview`'s smoke-check apparently never exercised deeply enough to catch: a `ko.i18n` registration-order bug in `js/const.js` that only breaks under Rollup's production chunking (dev server is unaffected). See `CHANGELOG.md` (2026-07-10) for full details on all of this. Lesson for future verification: "the build succeeds" and "the bundle loads via `vite preview`" are necessary but not sufficient checks — verify the *actual deployment artifact* (the Docker image, run the way it's really run) in a real browser before calling production done.
  - This also exposed and fixed a latent bug in `genversion`: the old `genversion -s js/version/version.js && r_js -convert js/version js/` pipeline produced an AMD `define()`-wrapped `js/version.js`, which the ESM app can't import (`"version" is not exported by "js/version.js"`) — this had been masked until now because `build/optimize.js` silently no-op'd before anything tried to actually load the bundle. Replaced with a single step: `genversion -e -s js/version.js`, which writes plain ESM (`export const version = ...`) directly — no intermediate directory, no `r_js -convert`, no `rimraf` needed.
  - `requirejs` was only ever used by `build/optimize.js` and by `r_js -convert` in the old `genversion` step — both gone now, so the `requirejs`, `@babel/*`, `terser`, and `html-document` devDependencies were dropped from `package.json` (not deferred to step 6 as originally guessed; turned out nothing else needed them).

**Tests: fixed — now on Node's built-in test runner.** `atlas/`'s suite is 18/18 passing (unchanged since 2026-07-07). Note this section originally tracked `atlas`'s tests only; `webapi/`'s separate `node --test` suite has grown alongside the bug-fixing above and stood at 47/47 as of 2026-07-08 end of day. Jest is gone (`jest.config.js` deleted, `jest` dropped from `package.json`). It had depended on a RequireJS shim (`requireAmd()`/`requirejsInstance`) to load AMD-style modules, which stopped working once app source moved to ESM (every test failed with `SyntaxError: Unexpected token 'export'`). `npm test` now runs `node --import ./tests/register-hooks.js --test`, and all 5 existing test files (`AutoBind`, `BemHelper`, `CommonUtils`, `DataTypeConverterUtils`, `ExceptionUtils`) were ported to plain ESM imports + `node:assert/strict`. `package.json` now declares `"type": "module"` (verified this doesn't break `vite build` or `vite --config`). The one module-resolution wrinkle — `CommonUtils.js` importing Vite-only bare aliases (`appConfig`, `atlas-state`, etc.) that plain Node can't resolve — is handled by a small custom resolve hook (`tests/hooks.js`, registered via `tests/register-hooks.js`) that redirects those specifiers to stub files in `tests/stubs/`. See `CHANGELOG.md` (2026-07-07) for details.

**Docs: `atlas/CLAUDE.md` updated 2026-07-09.** Previously stale since the "initial reformatting to neostandard" commit (`5b3d0360`, before all the AMD→ESM migration commits) — described the codebase as pure AMD/RequireJS with "no dev server". Now describes the ESM/Vite architecture: `vite.config.js` alias-based module resolution, `vite build` for production, `node --test` for the test suite, and the `NODE_ENV`-gated debug scaffolding.

**Lint: clean as of 2026-07-09.** `npm run lint` reports 0 problems. Three cleanup passes total:
- The `no-undef` category (248 errors) was fully cleared: real migration bugs (missing `$`/`_`/`saveAs`/`Prism`/`moment` imports left over from RequireJS's global shims, plus implicit-global-assignment bugs that only worked in non-strict AMD scripts), including genuine correctness bugs like `AuthAPI.isPermitted()` being silently broken for every permission check.
- `eslint --fix` plus a manual review pass cleared ~2300 more (pure formatting: indent, blank lines, tabs) and a further ~140 bug-shaped-but-mostly-harmless categories (`no-redeclare`, `no-unreachable`, `no-return-assign`, `no-fallthrough`) — the last one included one more real bug, a broken job-URL fallthrough in `JobDetailsService.js`.
- The remaining 1376 problems (`no-unused-vars` 438, `eqeqeq` 267, `camelcase` 177, `no-prototype-builtins` 139, `new-cap` 104, plus ~15 smaller rule categories) were cleared 2026-07-09 across 328 files by 7 parallel agents, each hand-fixing an assigned non-overlapping file list (not a blanket `--fix`, since several rules needed judgment — see below). All 18 tests still pass afterward. Found and fixed a few more real bugs along the way: a broken `!_.xor(...).length == 0` boolean-vs-number comparison in `multi-select.js`, an always-true `typeof x !== SomeClass` check in `FullAnalysis.js` (should've been `instanceof`), and a stray no-op `authApi.handleAccessDenied` expression (missing its call parens) in `negative-controls.js`. `eqeqeq` conversions preserved the deliberate `== null` idiom throughout (neostandard's "smart" eqeqeq already permits it, so nothing flagged needed that exception). `camelcase` renames were scoped to local vars/params only, never external API/DTO property names. `new-cap` false positives (lowercase-namespace library constructors like `new atlascharts.boxplot()`) got scoped `eslint-disable-next-line` comments instead of renames. One file, `js/extensions/plugins/css.min.js` (third-party minified RequireJS CSS-loader shim, not app source), was excluded from lint via `eslint.config.js` `ignores` rather than hand-edited — too risky to touch minified vendor code by hand.

**Debug scaffolding: kept permanently, gated on `NODE_ENV` as of 2026-07-09.** Rather than removing the module-load-failure diagnostics now that things feel stable, per user preference they're kept but gated so they produce zero output in production:
- `js/main.js` — the `log()` helper (writes each import step to `console.log`/`document.title`) and the top-level `run().catch(...)` failure handler are now wrapped in `if (process.env.NODE_ENV !== 'production')`.
- `index.html` — the `window.addEventListener('error', ...)` handler was converted to `<script type="module">` (required for Vite to statically replace `process.env.NODE_ENV` in inline HTML scripts — plain non-module inline scripts aren't processed by Vite's transform pipeline, confirmed by testing) and wrapped in the same guard.

Verified via a build-time probe: `NODE_ENV=production npm run build:dev` fully tree-shakes both blocks out of the bundle (zero occurrences of `[LOAD]`/`MODULE ERROR` in build output — not just runtime-skipped, esbuild's dead-code elimination removes them entirely), while `NODE_ENV=development` (or unset, Vite's dev-server default) keeps them and the dev server still boots cleanly. `npm run lint`, `npm test` (18/18), and a live smoke test all pass after the change.

**Legacy AMD-era cruft: removed 2026-07-09.** `js/settings.js` (the old RequireJS config file) turned out to be imported nowhere in the ESM app — dead since the migration completed, and it was the only thing still referencing several vendor/shim files, which made them look live when they weren't. Deleted together: `js/settings.js`; the RequireJS plugin shims `js/extensions/plugins/{text,css,css.min,less,optional}.js` (including the IE-era `css.min.js` CSS-loader polyfill — no IE support is expected anymore); the vendored `js/assets/d3-scale-chromatic.1.3.0.min.js` (superseded by the real npm package, already aliased in `vite.config.js`); and `js/assets/d3.slider.js` + `js/styles/d3.slider.css` (the slider widget was never actually instantiated by any component — verified by search, only its CSS was still imported by `main.js`). Also dropped the corresponding `vite.config.js` entries (the `text!`/`less!`/`css!` AMD-plugin aliases and the `d3-slider` alias) and the `js/extensions/plugins/css.min.js` eslint ignore. Verified via `npm run lint` (clean), `npm test` (18/18), `npm run build:dev` (succeeds), and a live dev-server smoke test (Home page loads with no console errors). Confirmed still genuinely in use and NOT touched: `jquery.ddslick.js`, `jnj.chart.js`, `knockout.selectOnFocus.js`, and the real npm `d3-scale-chromatic` package.

**`vite.config.js` — "Phase 3" referenced but not defined anywhere else (now moot).** A comment there used to read `// AMD plugin aliases (kept for transition; removed after Phase 3)`; those aliases are gone as of the cleanup above. No other doc ever defined what "Phase 3" consisted of, but everything it plausibly meant is now done: (1) build scripts run `vite build`, (2) old RequireJS build/test tooling is gone, (3) transitional aliases are dropped, (4) debug scaffolding is kept but production-gated (see above).

All items in the original "suggested next steps" list below are now done — the migration is complete. Kept as a record of how it got there.
